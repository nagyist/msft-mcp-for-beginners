# MCP stdio సర్వర్ - .NET పరిష్కారం

> **⚠️ ముఖ్యమైనది**: ఈ పరిష్కారం MCP స్పెసిఫికేషన్ 2025-06-18 ప్రకారం సిఫార్సు చేసిన **stdio ట్రాన్స్‌పోర్ట్** ఉపయోగించడానికి నవీకరించబడింది. అసలు SSE ట్రాన్స్‌పోర్ట్ డిప్రికేటెడ్ అయింది.

## అవలోకనం

ఈ .NET పరిష్కారం ప్రస్తుత stdio ట్రాన్స్‌పోర్ట్ ఉపయోగించి MCP సర్వర్‌ను ఎలా నిర్మించాలో చూపిస్తుంది. stdio ట్రాన్స్‌పోర్ట్ సులభం, మరింత సురక్షితంగా ఉంటుంది మరియు డిప్రికేటెడ్ SSE విధానంతో పోలిస్తే మెరుగైన పనితీరును అందిస్తుంది.

## ముందస్తు అవసరాలు

- .NET 9.0 SDK లేదా తరువాతి సంస్కరణ
- .NET డిపెండెన్సీ ఇంజెక్షన్ యొక్క ప్రాథమిక అవగాహన

## సెటప్ సూచనలు

### దశ 1: డిపెండెన్సీలను రీస్టోర్ చేయండి

```bash
dotnet restore
```

### దశ 2: ప్రాజెక్ట్‌ను బిల్డ్ చేయండి

```bash
dotnet build
```

## సర్వర్ నడపడం

stdio సర్వర్ పాత HTTP ఆధారిత సర్వర్ కంటే వేరుగా నడుస్తుంది. వెబ్ సర్వర్ ప్రారంభించకుండా, ఇది stdin/stdout ద్వారా కమ్యూనికేట్ చేస్తుంది:

```bash
dotnet run
```

**ముఖ్యమైనది**: సర్వర్ హ్యాంగ్ అయినట్లు కనిపిస్తుంది - ఇది సాధారణం! ఇది stdin నుండి JSON-RPC సందేశాలను ఎదురుచూస్తోంది.

## సర్వర్‌ను పరీక్షించడం

### పద్ధతి 1: MCP ఇన్స్పెక్టర్ ఉపయోగించడం (సిఫార్సు చేయబడింది)

```bash
npx @modelcontextprotocol/inspector dotnet run
```

ఇది:
1. మీ సర్వర్‌ను సబ్‌ప్రాసెస్‌గా ప్రారంభిస్తుంది
2. పరీక్ష కోసం వెబ్ ఇంటర్‌ఫేస్‌ను తెరుస్తుంది
3. అన్ని సర్వర్ టూల్స్‌ను ఇంటరాక్టివ్‌గా పరీక్షించడానికి అనుమతిస్తుంది

### పద్ధతి 2: డైరెక్ట్ కమాండ్ లైన్ పరీక్ష

మీరు ఇన్స్పెక్టర్‌ను నేరుగా ప్రారంభించి కూడా పరీక్షించవచ్చు:

```bash
npx @modelcontextprotocol/inspector dotnet run --project .
```

### అందుబాటులో ఉన్న టూల్స్

సర్వర్ ఈ టూల్స్‌ను అందిస్తుంది:

- **AddNumbers(a, b)**: రెండు సంఖ్యలను కలిపి జోడించండి
- **MultiplyNumbers(a, b)**: రెండు సంఖ్యలను గుణించండి  
- **GetGreeting(name)**: వ్యక్తిగతీకరించిన అభివాదం సృష్టించండి
- **GetServerInfo()**: సర్వర్ గురించి సమాచారం పొందండి

### Claude Desktop తో పరీక్షించడం

ఈ సర్వర్‌ను Claude Desktop తో ఉపయోగించడానికి, మీ `claude_desktop_config.json` లో ఈ కాన్ఫిగరేషన్‌ను జోడించండి:

```json
{
  "mcpServers": {
    "example-stdio-server": {
      "command": "dotnet",
      "args": ["run", "--project", "path/to/server.csproj"]
    }
  }
}
```

## HTTP/SSE తో ముఖ్య తేడాలు

**stdio ట్రాన్స్‌పోర్ట్ (ప్రస్తుత):**
- ✅ సులభమైన సెటప్ - వెబ్ సర్వర్ అవసరం లేదు
- ✅ మెరుగైన సెక్యూరిటీ - HTTP ఎండ్‌పాయింట్లు లేవు
- ✅ `WebApplication.CreateBuilder()` బదులు `Host.CreateApplicationBuilder()` ఉపయోగిస్తుంది
- ✅ `WithHttpTransport()` బదులు `WithStdioTransport()` ఉపయోగిస్తుంది
- ✅ వెబ్ అప్లికేషన్ బదులు కన్సోల్ అప్లికేషన్
- ✅ మెరుగైన పనితీరు

**HTTP/SSE ట్రాన్స్‌పోర్ట్ (డిప్రికేటెడ్):**
- ❌ ASP.NET కోర్ వెబ్ సర్వర్ అవసరం
- ❌ `app.MapMcp()` రౌటింగ్ సెటప్ అవసరం
- ❌ క్లిష్టమైన కాన్ఫిగరేషన్ మరియు డిపెండెన్సీలు
- ❌ అదనపు సెక్యూరిటీ పరిగణనలు
- ❌ ఇప్పుడు MCP 2025-06-18 లో డిప్రికేటెడ్

## అభివృద్ధి లక్షణాలు

- **డిపెండెన్సీ ఇంజెక్షన్**: సర్వీసులు మరియు లాగింగ్ కోసం పూర్తి DI మద్దతు
- **స్ట్రక్చర్డ్ లాగింగ్**: `ILogger<T>` ఉపయోగించి stderr కు సరైన లాగింగ్
- **టూల్ అట్రిబ్యూట్లు**: `[McpServerTool]` అట్రిబ్యూట్లతో క్లీన్గా టూల్ నిర్వచనం
- **అసింక్ మద్దతు**: అన్ని టూల్స్ అసింక్ ఆపరేషన్లను మద్దతు ఇస్తాయి
- **లోపాల నిర్వహణ**: సున్నితమైన లోపాల నిర్వహణ మరియు లాగింగ్

## అభివృద్ధి సూచనలు

- లాగింగ్ కోసం `ILogger` ఉపయోగించండి (నేరుగా stdout కు రాయవద్దు)
- పరీక్షించడానికి ముందు `dotnet build` తో బిల్డ్ చేయండి
- విజువల్ డీబగ్గింగ్ కోసం ఇన్స్పెక్టర్‌తో పరీక్షించండి
- అన్ని లాగింగ్ ఆటోమేటిక్‌గా stderr కు వెళ్తుంది
- సర్వర్ సున్నితమైన షట్‌డౌన్ సిగ్నల్స్‌ను హ్యాండిల్ చేస్తుంది

ఈ పరిష్కారం ప్రస్తుత MCP స్పెసిఫికేషన్‌ను అనుసరిస్తుంది మరియు .NET ఉపయోగించి stdio ట్రాన్స్‌పోర్ట్ అమలుకు ఉత్తమ పద్ధతులను చూపిస్తుంది.
